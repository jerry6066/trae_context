# 提交与 PR 规则

## Git 提交规范

- 提交信息使用祈使句，简要说明本次改动核心目的（如："add feature" 而非 "added feature"）。
- 提交标题限制在 50 字符以内，必要时添加详细描述（空一行后）。
- 一个提交聚焦一类改动，避免把不相关内容混在同一次提交中。
- 修复 bug 的提交应引用相关问题编号（如：Fixes #123）。

## PR 描述要求

PR 描述应包含以下核心内容：

- **背景**：说明为什么需要这个改动，解决什么问题
- **改动点**：列出主要修改的文件和逻辑
- **验证方式**：如何测试和验证改动（参考测试与验证规则）
- **潜在风险**：可能影响的功能模块或需要注意的地方

## 特殊变更处理

- **破坏性变更**：必须明确迁移方案、回滚方案或版本升级策略
- **配置或脚本更新**：补充执行示例、环境要求和验证步骤
- **依赖升级**：说明升级原因、版本变化、兼容性影响
- **性能优化**：提供基准测试数据或性能对比

## 代码审查准备

- 确保代码通过所有自动化检查（lint/test/build）
- 移除调试代码和无关的日志输出
- 更新相关文档（如 API 变更、使用说明等）
- 复杂逻辑添加必要的注释说明
